## 1. Blueprint References (ERP_Agentic_AI_Blueprint.md)

### Module 1: AI ERP Intelligence Agent
* **Section:** `ثانياً: الموديول الأول — AI ERP Intelligence Agent` (Lines 25 - 330)
* **Key Content:**
  * **Goal & Use Cases (Lines 27-55):** Four core domains: Financial, Operational, Invoice, and Tax.
  * **Dynamic Query Builder & Safety (Lines 78-118):** Three layers of protection (Read-Only DB User, SQL Templates, AST Verification). 80% template-driven strategy.
  * **Agent Workflow & Schema (Lines 131-191):** Single-orchestrator pattern using LangGraph. LangGraph state schema definition.
  * **Invoice Analysis (Lines 192-232):** Query-based direct table access on invoices, items, and vendors. Pattern detection rules.
  * **Tax RAG (Lines 233-247):** PDF rules retrieval with legal disclaimer.
  * **Adaptive Output Selector (Lines 248-290):** Layout decision table (text, metric, table, charts) based on result size and user intent.
* **Architectural Conclusions:**
  * Avoid any parallel multi-agent complexity.
  * DB connection pool must enforce read-only credentials for SQL queries generated by the LLM.
  * Avoid OCR and parsing documents; work purely with relational database models for invoice insights.

### Module 3: Customer Support / Issue Resolution Agent
* **Section:** `ثالثاً: الموديول الثالث — Customer Support / Issue Resolution Agent` (Lines 333 - 513)
* **Key Content:**
  * **Goal & Scope (Lines 335-371):** Customer identifier parsing and data fetcher tools.
  * **Orchestrator Flow & State Schema (Lines 372-443):** Parallel data fetching (Real invoice + Mock order/shipping/history) and review gate.
  * **Graceful Degradation (Lines 444-457):** Output fallback behaviors based on data completeness flags.
  * **Human-in-the-Loop Gate (Lines 458-476):** Rules for when manual review is mandatory (billing, refund, confidence < 0.7) and when it's optional.
* **Architectural Conclusions:**
  * Needs separate `/support` endpoint.
  * Must integrate mock data tables (`order_status`, `shipping`, `customer_history`) to run scenarios cleanly.
  * Mandatory Human-in-the-loop review mechanism is a hard requirement.

### System Architecture & Tech Stack
* **Section:** `خامساً: البنية التقنية الشاملة للنظام` (Lines 587 - 635) & `سادساً: Tech Stack الموصى به` (Lines 637 - 674)
* **Key Content:**
  * **Component Diagram (Lines 589-625):** React+TypeScript+shadcn/ui Frontend → FastAPI Backend → PostgreSQL + pgvector → OpenAI API (GPT-4o / GPT-4o-mini).
  * **Shared Services (Lines 627-635):** Authentication, LLM Client connection pool, SQL pool, logger, and error boundary.
* **Architectural Conclusions:**
  * FastAPI must host the LangGraph Agent Engine.
  * PostgreSQL with `pgvector` will house vector knowledge bases.
  * LLM client is a single initialization shared by all nodes.
