# شرح بنية وتدفق البيانات لوكيل الذكاء الاصطناعي (M1 Intelligence Agent)

## 1. نظرة عامة (Overview)
تم تصميم الوكيل **M1** ليعمل كـ **"محلل بيانات أعمال" (Business Data Analyst)** متطور لبيئة الـ ERP، متجاوزاً مفهوم الروبوت النصي التقليدي (Chatbot). 

يعتمد النظام على هندسة برمجية تُسمى **(Hybrid Stratified Routing with Bounded Analytical Execution)**. تضمن هذه البنية:
1. العزل بين **نية المستخدم** (مبيعات، فواتير، ضرائب) و**مستوى التعقيد** (استعلام مباشر، متابعة، تحليل معقد).
2. الأمان التام عبر تقييد التنفيذ بقواعد بيانات للقراءة فقط (Read-only) وفلاتر حماية (AST Validation).
3. استمرارية السياق المعرفي بين الأسئلة المتتالية.

---

## 2. الهيكل التقني للحالة (State Schema - `M1State`)
بنية البيانات التي تنتقل بين طبقات النظام (Nodes) مُعرّفة كـ `TypedDict` باسم `M1State`. تعتمد هندسة الـ LangGraph هنا على تمرير هذا الكائن بين العقد، وكل عقدة تقوم بتحديث أو إضافة بيانات جديدة.

إليك أبرز الحقول التقنية التي تنتقل من طبقة لأخرى:

```python
class M1State(TypedDict, total=False):
    # ── 1. Input (المدخلات) ──
    query: str                    # السؤال الأصلي
    language: Literal["ar", "en"] # اللغة المكتشفة تلقائياً
    session_id: str               # مُعرف الجلسة لربطها بقاعدة البيانات
    chat_history: list            # سجل المحادثة السابق

    # ── 2. Stratified Routing (التوجيه الطبقي) ──
    assigned_tier: Literal["T0", "T1", "T2", "T3", "T4", "T5", "T6"] # مستوى التعقيد
    domain_intent: str            # مجال السؤال (مثال: financial, sales, tax)
    router_reasoning: str         # تبرير الموجه لاختياره هذا المسار

    # ── 3. Structured Analytical Context (السياق التحليلي) ──
    analysis_frame: dict          # إطار التحليل الحالي (المقاييس، الفلاتر، التاريخ)
    prior_analysis_frame: dict    # إطار التحليل للسؤال السابق (للمتابعة)

    # ── 4. Query & Data (الاستعلام والبيانات العائدة) ──
    raw_data: list                # البيانات الخام من قاعدة البيانات
    data_confidence: float        # ثقة النظام في البيانات المستخرجة
    
    # ── 5. Evaluation & Enrichment (التقييم والإثراء) ──
    result_status: Literal["complete", "partial", "empty", "suspicious", "invalid"]
    anomaly_detected: bool        # هل يوجد شذوذ في الأرقام؟ (مثال: قفزة مفاجئة)
    anomaly_details: dict         # تفاصيل الشذوذ

    # ── 6. Output (المخرجات) ──
    output_format: OutputType     # شكل العرض (جدول، رسم بياني، بطاقة أرقام...)
    narrative: str                # السرد التحليلي (النص الذي يشرح الأرقام)
    chart_config: Optional[dict]  # إعدادات الرسم البياني (ECharts JSON)
    final_response: dict          # الرد النهائي المجمّع للمستخدم
```

## 3. تدفق البيانات التفصيلي (Data Flow Pipeline)

يبدأ التدفق من لحظة دخول `query` ويمر بالمراحل التالية:

### أ. مرحلة ما قبل التوجيه (Pre-Routing)
1. **`context_loader`**: يقوم بالاتصال بقاعدة البيانات لجلب `chat_history` وكذلك `prior_analysis_frame` (الـ Metadata للأسئلة السابقة).
2. **`intent_router`**: يستخدم الذكاء الاصطناعي لتحليل النص وتحديد الـ `domain_intent` والـ `assigned_tier`.
   - *مثال:* المستخدم يسأل "ما هي مبيعات شهر مايو؟" -> يحدد الـ Router أن `domain_intent: "sales"` و `assigned_tier: "T1"`.

### ب. مسارات التنفيذ (Execution Tiers)
بناءً على الـ `assigned_tier`، يتم تمرير الـ `M1State` إلى أحد المسارات السبعة:

| المسار | العقدة المسؤولة | الوظيفة ومثال توضيحي |
|---|---|---|
| **T0** | `t0_conversation` | **المحادثة العامة:** "مرحباً، من أنت؟". الرد يكون نصياً مباشراً دون استعلامات. |
| **T1** | `t1_dispatcher` | **الاستعلام عبر القوالب الجاهزة:** "أعطني إجمالي الفواتير المتأخرة". يتفرع إلى (`db_query`, `invoice_analysis`, `tax_rag`) لتنفيذ SQL موثوق مسبقاً. |
| **T2** | `t2_resolver` | **استعلام المتابعة (Follow-up):** "قارنها بالشهر السابق". يعتمد على الـ `prior_analysis_frame` لتنفيذ مقارنة أو فلترة جديدة. |
| **T3** | `t3_planner` / `t3_executor` | **التحليل المعقد (NL2SQL):** استعلام غير موجود بالقوالب. يقوم بتوليد SQL ديناميكي ويمر بحلقة حماية وتصحيح (Repair Loop) لضمان صحته وأمانه. |
| **T4** | `t4_clarification` | **الاستيضاح:** "كم كانت المبيعات؟" (لا يوجد تاريخ). يطلب الوكيل توضيح الفترة الزمنية. |
| **T5** | `t5_oos` | **خارج النطاق:** "ما هي حالة الطقس؟". يتم إبلاغ المستخدم بلباقة بأن السؤال خارج قدرات النظام. |
| **T6** | `t6_delegate_m3` | **التحويل للدعم الفني:** "شحنتي تأخرت". يتم تجهيز `m3_delegation_payload` لتمريره للوكيل M3. |

### ج. مرحلة التقييم والتنسيق (Evaluation & Formatting)
بالنسبة للمسارات التحليلية (T1 و T2 و T3)، بمجرد استخراج `raw_data` من قاعدة البيانات، يمر الـ `M1State` بخط الإنتاج التالي:

1. **`result_evaluator`**: يفحص الـ `raw_data`. هل هي فارغة؟ هل تجيب عن السؤال الفعلي؟ يتم تحديث `result_status`.
2. **`validation_enrichment`**: يمرر البيانات على خوارزميات بايثون نقية لاكتشاف الشذوذ المالي (Anomaly Detection). إذا ارتفعت التكاليف بنسبة 200%، يجعل `anomaly_detected = True`.
3. **`output_selector`**: بناءً على هيكل البيانات (عدد الأعمدة ونوعها)، يحدد أفضل `output_format` (مثلاً `line_chart` للبيانات الزمنية، أو `metric_card` للأرقام الفردية).
4. **`chart_config`**: إذا كان العرض رسماً بيانياً، يتم توليد إعدادات الـ Chart بشكل ذكي يتجنب اختيار الـ ID كمحور ويختار الأسماء والنصوص.
5. **`narrative_generator`**: هذه هي طبقة "المحلل". يُرسل الـ `raw_data` و تفاصيل الـ `anomaly` للذكاء الاصطناعي لكتابة سرد يشرح البيانات، مثلاً: *"بلغت المبيعات 50 ألف، نلاحظ ارتفاعاً ملحوظاً بسبب كذا..."*.

### د. مرحلة حفظ السياق (Context Saver & END)
أخيراً، جميع المسارات تمر على عقدة **`context_saver`**. هنا يتم حفظ الـ `analysis_frame` والـ `narrative` في الـ `conversations.metadata` داخل قاعدة البيانات (Supabase)، ثم تجميع `final_response` لإرسالها للواجهة الأمامية.

## 4. مثال تقني متكامل (End-to-End Example)

### الطلب الأول: "ما هي إيرادات شهر أبريل؟"
1. **`intent_router`**: يحدد `T1` ويحدد القالب المناسب للإيرادات.
2. **`t1_db_query`**: يُنفذ استعلام الـ SQL المخصص لشهر أبريل.
3. الـ `state` الآن يحتوي على: `raw_data = [{"month": "April", "revenue": 150000}]`.
4. **`output_selector`**: يقرر أن الأفضل هو `metric_card`.
5. **`narrative_generator`**: يكتب: *"بلغت إيرادات شهر أبريل 150 ألف دولار."*
6. **`context_saver`**: يحفظ الـ `analysis_frame` كالتالي:
   `{"metric": "revenue", "date_range": {"start": "2024-04-01", "end": "2024-04-30"}}`

### الطلب الثاني في نفس الشات: "قارنها بشهر مارس"
1. **`context_loader`**: يجلب الـ `analysis_frame` السابق للذاكرة السريعة `prior_analysis_frame`.
2. **`intent_router`**: يكتشف أنها متابعة ويوجه للمسار `T2`.
3. **`t2_resolver`**: يقرر نوع المتابعة `compare` ويحتاج لجلب بيانات مارس (عملية `requery`).
4. يُرسَل الطلب إلى **`t3_planner`**: يدمج الـ Metric القديم (revenue) مع الفلتر الجديد (شهر مارس) ويستخرج بيانات مارس.
5. البيانات تصبح: `[{"month": "March", "revenue": 120000}, {"month": "April", "revenue": 150000}]`.
6. **`output_selector`**: يغير العرض ديناميكياً إلى `bar_chart` أو `line_chart` للمقارنة.
7. **`narrative_generator`**: يكتب: *"نلاحظ زيادة في إيرادات أبريل (150 ألف) مقارنة بشهر مارس (120 ألف) بنسبة 25%."*
8. **`context_saver`**: يحدّث إطار التحليل ليشمل المقارنة استعداداً لأي سؤال قادم.
